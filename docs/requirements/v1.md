# 요구사항 명세 v1

> 출처: Issue #12 (`요구사항 명세`) 합의 결과 정리.
> 본 문서는 v1 도메인 어휘와 제약을 단단하게 박는 것이 목적이며, 구현 PR은 별도로 진행한다.
> 기존 설계 문서 `docs/superpowers/specs/2026-04-19-discord-membership-gate-design.md` 와 충돌하는 지점은
> 같은 문서 하단의 `v1 변경 사항 (Issue #12)` 섹션에 명시되어 있다.

## 0. 한 줄 정의

ScrimFinder는 **시간대와 티어가 맞는 두 LoL 5인 팀의 스크림 약속을 잡아주는** 매칭 서비스다.
승패·전적은 저장하지 않고, 매칭이 잡힌 시점에 양 팀에 Discord 봇이 알림을 보내는 데서 시스템의 역할은 끝난다.

## 1. 사용자 / 자격

- 로그인은 **Discord OAuth 단일**.
- **Discord 길드 멤버십은 *팀 합류*의 게이트.** 길드에 속하지 않은 사용자는 어떤 팀에도 합류할 수 없다.
  - 모집글 등록·매칭 신청·매칭 수락 시점에는 추가 검사 없음 (이미 합류 시 통과한 사람들만 그 흐름에 진입할 수 있기 때문).
  - 비로그인·비합류 사용자는 모집글을 *둘러볼* 수는 있다 (회원가입 ≠ 매 방문 로그인).
- **Riot Sign-On (RSO)** 은 *팀 합류 시점의 본인·티어 검증* 자리.
  - 현재 RSO는 가용하지 않으므로 임시로 **자기 신고 폼(Riot 닉네임 + 솔로 티어)** 으로 대체.
  - RSO가 살아나면 동일 자리를 RSO 결과로 교체한다. 도메인 자체는 바뀌지 않는다.

## 2. 팀 (Team)

- 한 사용자는 **N개 팀을 소유**할 수 있다.
  - 기존 `teams.unique (owner_user_id)` 제약과 `TEAM_ALREADY_EXISTS` 에러는 제거 대상.
- 한 사용자는 **N개 팀에 참여**할 수 있다 (소유와 무관).
- **팀 정원 = 정확히 5명.** 모집·매칭은 정확히 5명을 충족했을 때만 가능.
- v1 범위 밖: 팀 해체, 소유권 이전, 멤버 자진 탈퇴, 강퇴.

## 3. 멤버십 / 합류

- 합류 경로는 **초대 링크 단일**. 이전의 "익명 합류(`displayName`만 입력)" 경로는 폐기.
- 합류 절차:
  1. 초대 링크 클릭
  2. Discord 로그인 (필수)
  3. **Discord 길드 멤버십 검사** — 비가입자는 합류 거부
  4. RSO 가용 시: Riot 계정 연결 → PUUID·솔로 티어 자동 채움
     RSO 비가용 시: **자기 신고 폼 — Riot 닉네임 + 솔로 티어** 입력
  5. 활성(`ACTIVE`) 멤버 생성
- 같은 사용자가 같은 팀에 두 번 합류해도 멱등 (기존 동작 유지).
- 6번째 합류 시도는 `TEAM_FULL` 로 거부 (정원 5명 보장).

## 4. 모집글 (MatchPost)

- **공개 범위: 누구에게나 노출** (비로그인 포함).
- 핵심 필터: **시간대, 티어 범위 (`minTier`–`maxTier`)**. 그 외(설명, 인원 등)는 부수.
- **OPEN 등록 가드**: 팀의 `ACTIVE` 멤버가 **정확히 5명**일 때만 가능.
  - 현재 `registerMatchPostAction` 의 "≥1명" 검사는 "==5명" 으로 강화한다.
- 한 팀당 동시 `OPEN` 모집글은 1개 (`OPEN_MATCH_ALREADY_EXISTS`) — 유지.
- 자동 종료 (둘 다 lazy, 별도 cron 없음):
  - 매칭이 확정되면 `CLOSED`.
  - `availableTime` 이 *과거가 된 채 조회/매칭 잡에 닿는 순간* `CLOSED`.
- 소유자가 직접 취소하는 `CANCELLED` 액션은 별건 (v1 범위 밖).

## 5. 매칭 (Match) — 신규 엔터티

두 팀을 묶는 신규 엔터티 `Match` 를 도입한다. 승패·기록은 저장하지 않고, *매칭 확정 = 양 팀에 Discord 봇 알림이 송신된 시점*.

### 5.1 흐름 (수동 + 자동 병행)

수동 신청과 자동 매칭은 **항상 병행**한다. 모든 `OPEN` 모집글은 동시에:
- 자동 매칭 잡의 후보이자
- 수동 신청 대상이다.

흐름은 두 갈래:

- **수동 신청**
  1. 다른 팀이 모집글에 신청
  2. 모집팀 소유자가 수락
  3. `Match` 생성 + 양 팀의 `OPEN` 모집글 모두 `CLOSED`
  4. Discord 봇이 양 팀에 알림 송신

- **자동 매칭**
  1. 시간대·티어가 맞는 `OPEN` 모집글 두 개를 시스템이 짝지어 후보로 띄움
  2. 양 팀이 수락
  3. `Match` 생성 + 둘 다 `CLOSED`
  4. Discord 봇이 양 팀에 알림 송신

### 5.2 티어 비교 규칙

- **팀 대표 티어 = 5명의 평균.** (티어를 ord 매핑해 평균 계산.)
- **자동 매칭**: 양 팀의 평균 티어가 *상대 모집글의 `minTier`–`maxTier` 범위 안에* 들어와야 매칭.
- **수동 신청**: 시스템은 강제 검증하지 않는다. 화면에 신청 팀의 평균 티어와 멤버 티어 분포를 노출 → 모집팀 소유자가 직접 보고 결정.

## 6. 라이프사이클 / 정리

- `MatchPost`: lazy `OPEN → CLOSED` 만으로 충분.
- `InviteLink`: 사용 시점에만 만료 검사 (cron 불필요).
- 사용자 탈퇴: 모든 관련 데이터 삭제. 보존 의무 없음 (개인정보·법적 페이지 부담도 같이 줄어듦).

## 7. 코드 상의 주요 변경 윤곽 (구현 PR을 위한 가이드)

후속 PR에서 닿게 될 코드 자리를 미리 적어둔다. 명세 자체는 위 §1–§6 으로 충분하다.

### 도메인 (`src/shared`)

- `src/shared/types/core.ts`
  - `Match` 신규: `{id, leftPostId, rightPostId, leftTeamId, rightTeamId, origin: "MANUAL" | "AUTO", confirmedAt}`
  - `MatchProposal` (또는 `MatchRequest`) 신규: 상태 `PENDING | ACCEPTED | REJECTED | WITHDRAWN`. 자동 매칭 후보도 같은 자료형으로 표현.
  - 멤버 자기 신고 필드: `TeamMember` 또는 `User` 에 `riotGameName?`, `riotTagLine?`, `soloTier?` 위치 결정 필요 (v1 명세상으로는 *합류 시 입력*이므로 `TeamMember` 쪽이 자연스러움).
- `src/shared/constants/`
  - `tierOrder`: `LolTier` → 정수 매핑. 평균 계산과 범위 비교의 단일 출처.
- `src/shared/types/actions.ts`
  - 신규 에러 코드: `TEAM_FULL`, `DISCORD_GUILD_MEMBERSHIP_REQUIRED`, `RIOT_PROFILE_REQUIRED` (자기 신고 폼 미입력), 매칭 흐름용 `MATCH_PROPOSAL_NOT_FOUND`, `MATCH_ALREADY_CONFIRMED` 등.
  - 제거 또는 무력화: `TEAM_ALREADY_EXISTS` (한 사용자 다중 팀 소유 허용).

### 정책 / 가드 (`src/server`)

- `src/server/actions/teams.ts` — `TEAM_ALREADY_EXISTS` 분기 제거, 소유자 자동 ACTIVE 멤버 생성도 v1 합류 흐름에 맞춰 정리 (소유자도 합류 절차를 거쳐야 함, 설계 문서 §Team Creation 그대로).
- `src/server/actions/invite-links.ts` — `joinTeamByInviteAction` 에서 익명 합류 분기 제거, **Discord 길드 멤버십 검사** 호출 추가, **자기 신고 폼 입력** 필수화, `TEAM_FULL` 가드 추가, `usedCount` 원자 증가.
- `src/server/actions/matches.ts` — `registerMatchPostAction` 의 "ACTIVE ≥ 1" 가드를 "ACTIVE == 5" 로 강화. (길드 게이트는 합류 시점으로 옮기므로 여기서는 부르지 않음.)
- `src/server/actions/match-proposals.ts` (신규) — 수동 신청 / 자동 매칭 후보의 생성·수락·거절. 수락 시 `MatchPost.OPEN → CLOSED` 는 *조건부 update* 로 race-safe 하게.
- `src/server/services/discord-bot.ts` — 매칭 확정 알림 송신 함수 추가.
- `src/server/jobs/auto-match.ts` (신규) — 시간대·티어가 맞는 `OPEN` 모집글 페어링 잡.

### 인프라

- `supabase/schema.sql`
  - `teams` 의 `UNIQUE (owner_user_id)` 제거.
  - `team_members` 에 `riot_game_name`, `riot_tag_line`, `solo_tier` 컬럼 추가 (또는 `users` 에 추가, 위치는 후속 PR에서 결정).
  - `matches`, `match_proposals` 테이블 신규.
- `/api/auth/riot/callback` (`src/app/api/auth/riot/callback/route.ts`)
  - RSO 가용 전까지 임시 라우트라는 점을 코드 코멘트로 명시. 가용해지면 진짜 RSO 콜백으로 회수한다.

## 8. 잠재적 부채 (지금은 안 해도 되지만 적어둠)

- 합류 시점에만 길드 검사를 하므로 **합류 후 길드 탈퇴**한 사용자에 대한 알림은 봇이 실패할 수 있다. 매칭 알림 직전에 한 번 더 검사하는 방어선은 추후 추가.
- 자동 매칭 잡과 동시 수동 수락이 충돌할 수 있으므로 `MatchPost.status` 갱신은 *조건부 update* (`WHERE status = 'OPEN'`) 로.
- 초대 링크 `usedCount` 동시성도 같은 김에 원자 증가로 정리.
- 구버전 익명 합류 데이터가 Supabase 에 쌓여 있다면 마이그레이션 정책이 필요 (현재 in-memory 가 기본이라 우선 무시 가능).
- `MatchPost.CANCELLED` 소유자 액션, 매칭 노쇼·재매칭 흐름.
- 멤버 역할 확장 (코치, 서브, 매니저 등).

## 9. 추후 확장 시점 (v1 범위 밖)

- 팀 해체 / 소유권 이전 / 강퇴.
- 매칭 결과(승패) 기록.
- 서버 지역, 챔프 풀, 연패 보정 등 매칭 정밀도 향상.
- 길드/봇 알림 실패 시 재시도·DLQ.
